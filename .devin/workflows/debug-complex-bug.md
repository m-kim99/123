## 사전 조건
- 같은 버그를 2회 이상 수정 시도했지만 해결되지 않았을 때
- 에러의 원인이 불명확하거나 여러 파일에 걸쳐있을 때
- 수정 후 다른 곳에서 부작용이 발생할 때

## Step 1: 이전 시도 정리

수정을 시도했지만 효과 없었던 파일/라인을 **배제 목록**으로 정리한다.
이 목록에 있는 곳은 원인이 아닌 것으로 간주하고 탐색에서 제외한다.
사용자에게 이전 시도 내용을 확인한다.

## Step 2: 아키텍처 및 스키마 문서 읽기
// turbo

`docs/ARCHITECTURE.md`를 읽는다. 특히 **섹션 5. 암묵적 의존성 목록**을 반드시 확인한다.
현재 버그 증상과 관련될 수 있는 암묵적 의존성이 있는지 체크한다.

DB 관련 에러(RLS, FK, 권한, 빈 응답 등)가 의심되면 `docs/schema.sql`에서
해당 테이블의 실제 스키마, RLS 정책, 트리거, 함수를 확인한다.
코드의 쿼리와 실제 DB 스키마가 일치하는지 대조한다.

## Step 3: 데이터 흐름 전체 매핑 (수정 금지)

**이 단계에서는 코드를 절대 수정하지 않는다.**

문제 동작과 관련된 데이터 흐름을 처음(사용자 액션)부터 끝(화면 렌더링)까지 추적한다.
다음을 포함해야 한다:
- 트리거된 함수와 호출 체인
- 변경되는 상태(Zustand store)
- 외부 시스템 호출 (Supabase Auth, DB, R2, Realtime)
- 이벤트 리스너 (onAuthStateChange, MutationObserver 등)

## Step 4: 암묵적 의존성 체크리스트

아래 항목을 현재 버그와 대조하여 확인한다:

- [ ] **Supabase Auth 이벤트 체인**: signIn/signOut이 onAuthStateChange를 통해 다른 store에 영향을 주는가?
- [ ] **checkSession 자동 생성**: users 테이블에 없는 계정이 자동 생성되거나 강제 로그아웃되는가?
- [ ] **Realtime 채널**: Auth 세션 변경으로 Realtime 구독이 끊기는가?
- [ ] **R2 파일 경로**: DB 레코드와 실제 R2 파일 경로가 불일치하는가?
- [ ] **CSS 다크모드 연쇄**: index.css의 !important 오버라이드가 컴포넌트 스타일을 덮어쓰는가?
- [ ] **RLS 정책**: 코드는 정상이지만 DB 정책이 쿼리를 차단하는가?
- [ ] **구독 제한**: subscription.ts의 제한 체크와 RLS가 이중으로 차단하는가?
- [ ] **Capacitor 분기**: 웹/네이티브 환경 차이로 인한 문제인가?

## Step 5: 원인 후보 나열

가능한 원인을 **최소 3가지** 나열하고, 각각:
1. 어떤 코드가 근거인지 파일과 줄 번호를 인용
2. 이 원인이 맞다면 어떤 증상이 나타나야 하는지 예측
3. 실제 증상과 일치하는지 판단

사용자에게 후보 목록을 보여주고 확인을 받는다.

## Step 6: 로그 삽입 (수정 금지)

확인된 후보 지점에 `console.log`를 추가한다.
- 입력값, 출력값, 분기 조건을 출력
- **버그 수정 코드는 이 단계에서 넣지 않는다**
- 사용자에게 로그를 확인하고 결과를 알려달라고 요청한다

## Step 7: 로그 결과 기반 수정

사용자가 제공한 로그 결과를 바탕으로:
1. 원인 범위를 확정
2. **최소한의 수정**만 적용 (다른 흐름에 부작용이 없는지 확인)
3. 삽입했던 console.log를 제거
4. 수정 사항을 사용자에게 설명

## 주의사항

- 사용자가 지목한 파일이 실제 원인이 아닐 수 있다. 사용자의 가정에 동의하지 말고 코드를 직접 확인하라.
- 증상을 패치하지 말고 근본 원인을 해결하라.
- 한 번에 여러 곳을 수정하지 말고, 한 지점씩 수정하고 검증하라.
